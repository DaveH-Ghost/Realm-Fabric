## 0.6.0a ‚Äî Movement blocking + pathfinder ‚úÖ

### Engine

- **`src/occupancy.py`** ‚Äî tile enterability, `movement_exceptions`, `resolve_standable_goal` (nearest standable tile when goal is blocked)
- **`src/pathfinding.py`** ‚Äî BFS pathfinding (8-way, cost 1 per step, same 5e rules as before); `walk_with_pathfinding` returns final position, reached flag, and path steps
- **`Object.blocks_movement`** ‚Äî default **`true`** (objects block their tile)
- **`Object.movement_exceptions`** ‚Äî entity ids allowed to pass through / stand on that object‚Äôs tile
- **`Agent.blocks_movement`** ‚Äî default **`false`** (agents can share tiles, D&D-friendly)
- **`Agent.movement_exceptions`** ‚Äî same exception list semantics for blocking agents
- **`move()`** ‚Äî limited `move_speed` uses pathfinder; unlimited move teleports only to **enterable** tiles (cannot stand on a blocking object‚Äôs tile unless excepted)
- Move to **entity id** resolves to the object/agent tile, then paths to nearest **standable** tile when that tile blocks movement

### CLI

```
create-object ... [blocks-movement true|false] [movement-exception id1,id2]
edit-object <id> ... blocks-movement false movement-exception agent_01
create-agent ... [blocks-movement true|false] [movement-exception ...]
edit-agent <id> ... blocks-movement true movement-exception obj_door_01
```

### Save / snapshot

- Object and agent JSON include `blocks_movement` and `movement_exceptions`
- Still **`snapshot_version: 2`** (additive fields; older saves default objects to blocking, agents to non-blocking)

### Tests

- `tests/test_movement_blocking.py`
- `tests/test_move_pathing.py`, `tests/test_move_target.py`, `tests/test_session.py` updated for blocked tiles

### realm-studio (0.6.0a)

- **Create object** / **Edit object** ‚Äî **Blocks movement** checkbox (default on); **Movement exceptions** comma-separated entity ids when blocking is enabled
- Removed **Add eat action** checkbox from create object (use **Manage actions‚Ä¶** instead)

### Breaking / behavior changes

- Agents **no longer teleport onto** blocking object tiles when moving to `obj_*` targets; they stop on the nearest standable adjacent tile
- Greedy `walk_towards` in `src/pathing.py` remains for unit tests; **runtime move** uses BFS when `move_speed` is set
- Cookie-style demos that need same-tile interact (`range 0`) should use **`blocks-movement false`** on the object

### Move result clarity

- **Coordinate partial moves** ‚Äî first-person result includes stop tile: `You moved towards (4, 4), stopping at (3, 2).`
- **Entity partial moves** ‚Äî no coordinates; distance only: `You moved towards Ceramic Ball; you are still 2 steps away.`
- **Blocking entity arrival** ‚Äî adjacent stop counts as success: `You have successfully moved next to Ceramic Ball.`
- **Already adjacent** ‚Äî `You are already next to Ceramic Ball.` (not the object‚Äôs tile coordinate)
- **Blocked coordinate** ‚Äî `You are already at (1, 1), as close as you can get to (2, 2).`
- **Unreachable entity** ‚Äî names the blocker: `You cannot reach Ceramic Ball; Wall is blocking the way.`
- **`src/occupancy.py`** ‚Äî `find_blocker_between()` for blocker identification on failed paths

### Player agents (realm-studio testing)

- **`Agent.is_player`** ‚Äî human-controlled agent; same simulation rules, no LLM on turn
- **CLI:** `create-agent ... player true`, `edit-agent <id> player true|false`
- **Snapshot / save:** `is_player` on agent JSON (default `false`)
- **realm-studio:** **Player** checkbox on create/edit agent; **Run turn ‚ñ∂** opens manual compound-turn form instead of calling the LLM
- **`POST /api/turn/manual`** ‚Äî validate and run `AgentCompoundTurn` JSON for player agents

## 0.6.0b ‚Äî Interact pathing (overwrites move) ‚úÖ

### Engine

- **`interact()`** ‚Äî if out of `action.range`, spends **`move_speed`** pathing toward the nearest enterable tile from which the action is legal, then executes if in range
- Unlimited **`move_speed`** (`None`) teleports to that interact tile when pathing for interact
- **`execute_nav_phase`** ‚Äî when compound turn `action` is **`interact`**, the explicit **`move`** field is **ignored** (no separate nav step); pathing happens inside interact

### Compound turn semantics

- Order unchanged in record: **move step omitted** ‚Üí look ‚Üí speak ‚Üí interact/emote
- LLM may still send `move` + `interact` in one JSON turn; engine uses interact-path only
- **Interact pathing** records a **move** turn step (move-style result/passive) when pathing changes the agent's tile, so memory includes where they stopped

### Tests

- `tests/test_interact_pathing.py`
- `tests/test_object_actions.py` updated (far cookie + `move_speed 1` for out-of-range cases)

### Notes for smoke testing

- Kick ball from distance: one compound turn with `action: interact` should path adjacent (if `move_speed` allows) and kick without a separate move
- `step-compound 2,3 interact obj_‚Ä¶ eat` ‚Äî move token is ignored when interact is present; agent paths via interact rules only
