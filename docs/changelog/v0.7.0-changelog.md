## Why 0.7.0

After 0.6.x, the simulation stack is solid (grid, triggers, handlers). Downstream projects can depend on `campaign-rpg-engine` and run turns programmatically, but:

- **`campaign_rpg_engine` exports are incomplete** — lorebooks, prompt blocks, and memory-module types still require `src.*` imports (as campaign-rpg-studio does today).
- **World editing is CLI-shaped** — `session.run_command("create-object ...")` parses strings; campaign-rpg-studio’s frontend builds those strings for GM tools. That is fine for the **reference editor**, not for product apps.
- **No thin HTTP template** — campaign-rpg-studio is a full GM UI, not a minimal “embed the engine” server.
- **Install / semver story** is informal — path dependencies from the monorepo only.

**0.7.0** delivers the runway. **0.7.1+** on this repo addresses integration gaps; the **demo app** proves the pattern from its own repository.

## Design principles

### Apps use typed Python, not CLI strings

| Layer | App code (0.7.0+) | CLI / campaign-rpg-studio GM |
|-------|-------------------|------------------------|
| Turns | `session.run_compound_turn(AgentCompoundTurn(...))` | `run`, agent name, `step-compound` |
| World edits | `session.create_object(...)`, `session.edit_agent(...)`, etc. | `create-object`, `edit-agent`, … |
| Debug / human | Optional `run_command` in tests or REPL | `realm` stepper, `/api/command` |

`run_command` remains for the **`realm` CLI** and optional **debug HTTP** endpoints. It should **delegate** to the same typed methods apps use.

### `campaign_rpg_engine` vs `src.*`

- **`campaign_rpg_engine`** — semver-stable surface for apps.
- **`src.*`** — engine internals; CLI and tests may import; **not** guaranteed stable.

Document what breaks in patch vs minor releases.

### campaign-rpg-studio is not the template

campaign-rpg-studio stays the **kitchen-sink GM reference**. New apps start from **`building-on-campaign-rpg-engine.md`** + **`examples/minimal-server/`**, not by forking studio.

## 0.7.0b — Typed world-editing API

### `Session` methods (proposed)

Wrap existing `area_edit` / `session_area_edit` validation — **same rules as CLI**, structured arguments:

```python
# Illustrative — exact signatures TBD in implementation

session.create_object(
    name="Console",
    position=(3, 4),
    *,
    area_id=None,  # default active area
    passive_description="...",
    description="...",
    actions={...},  # or add_action later
    handler_id=...,
    width=1,
    height=1,
    hidden=False,
    blocks_movement=True,
) -> SessionResult  # message + created id in result or returned Object ref

session.edit_object(object_id, **fields) -> SessionResult
session.delete_object(object_id) -> SessionResult
session.create_agent(name, position, *, personality="", memory_module="recent_turns", is_player=False, ...) -> SessionResult
session.edit_agent(agent_id, **fields) -> SessionResult
session.delete_agent(agent_id) -> SessionResult
session.add_object_action(object_id, action: ObjectAction) -> SessionResult
session.remove_object_action(object_id, action_name) -> SessionResult

session.create_area(area_id, *, description="", width=5, height=5) -> SessionResult
session.edit_area(area_id, **fields) -> SessionResult
session.delete_area(area_id) -> SessionResult
```

### Behaviour

- ID generation, footprint validation, handler param validation — same as CLI.
- Multi-area: update `agent_area` when creating agents; respect `active_area_id` for GM-scoped edits.
- Return **`SessionResult`** / **`CommandResult`** with clear `ok` + `message` (consistent with existing session methods).

### Tests

- Parity tests: typed API vs equivalent CLI for representative cases (port from `test_stepper` / `test_area_edit`).
- Apps never **required** to call `run_command` for world setup.
