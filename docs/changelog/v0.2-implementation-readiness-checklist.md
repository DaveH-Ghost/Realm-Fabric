# Version 0.2 Implementation Readiness Checklist

**Purpose:**  
This checklist defines the key design decisions, scope, and agreements for V0.2. It builds on the shipped V0.1 foundation (multi-agent passive observation, world editing, `passive_result`, agent `pdesc`/`desc`/`personality`, one structured action per LLM call per agent turn).

**Rule:** Checklist items are agreed before implementation. **V0.2 is complete** — see status footer for as-shipped notes (`v0.2.0`).

**Project Scope (Sections 0–4 implemented and shipped in v0.2.0):**
- Builds on V0.1: same 5×5 grid, ManualStepper + rich logging, per-agent memory, mutual passive vision, runtime world editing.
- **Coordinate-based move:** agents target `(x, y)` and teleport there (in-bounds validation only; no pathing or blockers in V0.2).
- **Compound agent turns:** an agent may perform multiple steps in one turn — move (optional), then look(s) (optional), then one **turn action** (`speak` or object interact in V0.2). Requires **at least two LLM calls** per agent turn (navigation phase, then action phase with post-move perception).
- **Object custom actions:** objects expose declarative interact actions (e.g. `eat` on a cookie) with range checks and fixed effect handlers.
- Fully manual control preserved: human still decides which agent acts and when (`switch`, `run`, typing agent name).
- **Memory (V0.1 unchanged for V0.2):** last 10 `TurnRecord`s, `looked_at` / `ever_looked`, single latest `passive_result`. Compound turns should log in an **event-friendly** shape for V0.2.5 — no memory manager, beliefs/goals DB, or tiered policies in V0.2.
- **Proposed out of scope for V0.2:** full memory subsystem (deferred to **V0.2.5** — see [ROADMAP.md](../ROADMAP.md)), pathing algorithms, blocker objects, multi-tile occupancy, speak targeting, relationships/beliefs graph, agent-initiated world edits, automatic turn sequencing, GUI (V0.3), arbitrary scripting / freeform command execution on objects, hearing range / line-of-sight, new third-party dependencies.
- **Goal:** Richer turns and object interaction while keeping the prompt → reasoning → action → memory/perception loop observable and testable.

See also: [ROADMAP.md](../ROADMAP.md), [v0.1-implementation-readiness-checklist.md](v0.1-implementation-readiness-checklist.md), [LONG_TERM_GOALS.md](../../LONG_TERM_GOALS.md).

**Agreed implementation order:**
1. Coordinate-based move — isolated schema/perception change
2. Compound turns (two-phase LLM) — event-friendly `TurnRecord` for V0.2.5 hooks
3. Object custom actions — depends on action-phase schema
4. Cross-cutting (Section 4) — woven in during implementation; docs and `v0.2.0` tag at ship time

## Section 1: Coordinate-based move

**Depends on:** V0.1 movement and `passive_result` behavior.

- [x] Replace cardinal one-step move with coordinate targeting

  **Agreed Decision for V0.2:**

  **Target format**
  - **Canonical (advertised in prompts, few-shots, help, manual `step`):** comma-separated integers **`"x,y"`** (e.g. `"2,3"`). Parser also accepts optional whitespace: `"2, 3"`.
  - **Also accepted (parser/validator only — not advertised):** parenthesized forms **`"(x,y)"`** and **`"(x, y)"`**, normalized by stripping outer parens before parse. Reduces LLM formatting failures when the model mirrors passive-vision coordinate notation.
  - Reject non-coordinates: cardinals, object ids, wrong arity (`"2"`, `"2,3,4"`), non-integers → `ERR:INVALID_TARGET` (or `ERR:INVALID_COORDINATES` if split in Section 4).

  **Cardinal directions**
  - **Remove** `north` / `east` / `south` / `west` entirely in V0.2 (no deprecated alias).

  **Validation (V0.2)**
  - Parsed coordinates must be integers in **0–4** inclusive.
  - No pathing, blockers, or tile-occupancy checks.

  **Semantics**
  - **Teleport** directly to `(x, y)`; intermediate tiles are not visited and do not affect perception.

  **Same-tile move**
  - **Allowed** as success (see result strings below).
  - **No** `passive_result` update (nothing observable changed for other agents).

- [x] Define move `result` and `passive_result` strings

  **Agreed Decision for V0.2:**

  | Outcome | First-person `result` | Third-person `passive_result` |
  |---------|----------------------|------------------------------|
  | Success (new position) | `You moved to (x, y).` | `{name} moves to (x, y).` |
  | Same tile (no-op) | `You are already at (x, y).` | *(empty — no update)* |
  | Off grid | `You tried to move to (x, y), but that is outside the room.` | *(empty)* |
  | Malformed / unparsable target | `This action wasn't recognized, ERR:INVALID_TARGET, …` | *(empty)* |

  First-person uses past tense (`moved`) to match V0.1; passive stays present (`moves`) per Section 0.

- [x] Update perception and prompt for coordinate move

  **Agreed Decision for V0.2:**

  **System rules (`prompt.py`)**
  - Replace one-tile cardinal wording with: move to any in-bounds grid coordinate `(x, y)` in one action (teleport within the room). Document target as **`"x,y"`** only (not parenthesized form).

  **Available actions block**
  - Do **not** enumerate all 25 tiles.
  - Always include:
    - `You are at (x, y).`
    - `You may move to any coordinate (x, y) where x and y are integers from 0 to 4.`

  **Few-shot examples**
  - Replace all cardinal move examples with coordinate targets (e.g. `"target": "2,4"`).

  **Schema / field docs**
  - `target` for `move`: in-bounds grid coordinate string, documented as `"x,y"`.
  - Pydantic validator uses shared coordinate parser (accepts silent `"(x,y)"` variants).

  Section 2 `AgentNavigationTurn` / `move_target` uses the same `"x,y"` format and parser.

- [x] Manual stepper parity

  **Agreed Decision for V0.2 (as implemented):**

  - **Syntax:** coordinate move via `step-compound 2,3 …` or debug-only `step-nav 2,3` (canonical `"x,y"`). Parser accepts silent `"(x,y)"` variants.
  - **Behavior:** Same execution path as LLM move (`move()` in `src/actions/move.py`).
  - **Help / intro:** Document coordinate move; cardinal direction commands removed.
  - Isolated nav/action debug: `step-nav`, `step-action` (Section 2).

- [x] Future-proofing note for pathing (document only, not implemented)

  **Agreed Decision for V0.2:**

  - All move side effects go through **one function** in `src/actions/move.py` — `move(agent, world, target: str) -> ActionOutcome` — called from simulation, stepper, and (later) navigation phase.
  - V0.2: parse/normalize target → validate in-bounds → set `agent.position`.
  - **Documented limitation:** no path, no blockers, no “tiles passed through” updates; pathing plugs in here in a later version without changing the turn model.

- [x] Success criteria for Section 1

  **Agreed Success Criteria for V0.2 (Section 1):**

  | # | Test | Expected result |
  |---|------|-----------------|
  | 1 | Move to valid `(2,3)` from `(1,1)` | Position `(2,3)`; `You moved to (2, 3).`; `passive_result` `{name} moves to (2, 3).` |
  | 2 | Move to `(5,5)` or `(-1,0)` | `ERR:INVALID_COORDINATES`; position unchanged; no `passive_result` update |
  | 3 | Move to current tile | `You are already at (x, y).`; position unchanged; no `passive_result` update |
  | 4 | Targets `"2,3"`, `"2, 3"`, `"(2,3)"`, `"(2, 3)"` | All succeed equivalently (parser normalization) |
  | 5 | Malformed `"north"` or `"2"` | Validation or runtime failure; no position change |
  | 6 | Other agent `vision` after successful move | Agent line shows new coordinates + updated `passive_result` |
  | 7 | `step-compound 2,3` (or `step-nav 2,3`) | Same move semantics as LLM nav phase |
  | 8 | Prompt / few-shots | Cardinal move instructions removed; advertised target form is `"x,y"` only |

## Section 2: Compound turns (multiple steps per agent turn)

**Depends on:** Section 1 (coordinate move).

- [x] Agree the execution order for one agent turn

  **Agreed Decision for V0.2:**

  Fixed pipeline per **agent turn** (one `turn_number`):

  1. **Navigation phase** (LLM) → optional **move** execute
  2. **Action phase** (LLM) → optional **look** (at most one) → optional **turn action** (at most one)

  | Rule | Decision |
  |------|----------|
  | Skip phases | **Yes** — any subset allowed (e.g. look + speak without move; move only; speak only) |
  | Multiple looks | **No in V0.2** — at most **one** `look_target` per action phase |
  | Turn action | **At most one** of `speak`, `interact`, or `none` (interact per Section 3) |
  | `speak` + `interact` same turn | **No** |

- [x] Agree LLM call architecture

  **Agreed Decision for V0.2:**

  | Call | When | Prompt state | Output |
  |------|------|--------------|--------|
  | **Navigation** | Always on LLM `run` / typing agent name | Pre-turn passive vision + move rules | `move_target: "x,y"` or `null` (stay) |
  | **Action** | Always after nav phase completes (move executed or skipped) | **Post-move** passive vision + look rules + interact list (Section 3) + speak | `look_target`, `turn_action`, fields per type |

  - **Always two LLM calls** per agent turn on the LLM path (`null` move allowed).
  - **No third call** for look re-planning in V0.2.

  **Failure policy:**

  | Failure | Behavior |
  |---------|----------|
  | Navigation LLM invalid / parse error | **Abort entire agent turn** — no move, no action phase, `session_turn` does not increment |
  | Navigation valid, move fails (off-grid) | **Proceed to action phase** from unchanged position; failure in composite result |
  | Action LLM invalid / parse error | **Abort action phase** — nav/move already committed; partial turn recorded |
  | Action valid, look fails (not visible) | Look error in composite result; **turn action still runs** if specified |
  | Action valid, turn action fails | Error in composite result; move/look successes kept |

- [x] Define structured output schemas (navigation + action)

  **Agreed Decision for V0.2:**

  **Two separate Pydantic models** on the LLM compound path (replace single-action `AgentTurn`):

  ```python
  class AgentNavigationTurn(BaseModel):
      reasoning: str
      move_target: Optional[str]  # "x,y" or null; Section 1 parser (silent "(x,y)" OK)
      confidence: Optional[str]
      emotion: Optional[str]

  class AgentActionTurn(BaseModel):
      reasoning: str
      look_target: Optional[str]
      turn_action: Literal["speak", "interact", "none"]
      target: Optional[str]       # object id when interact
      action_name: Optional[str]  # e.g. "eat"
      content: Optional[str]      # speak text
      confidence: Optional[str]
      emotion: Optional[str]
  ```

  **Validation:**
  - `move_target`: Section 1 coordinate parser when non-null.
  - `turn_action == "speak"` → `content` required; existing speak limits apply.
  - `turn_action == "interact"` → `target` + `action_name` required (Section 3).
  - `turn_action == "none"` → `content`, `target`, `action_name` empty/null.
  - `look_target` visibility: runtime in action phase, not Pydantic.

- [x] Define turn execution engine (`step_agent_turn` or equivalent)

  **Agreed Decision for V0.2:**

  - Orchestrator: **`step_agent_turn(agent, world, nav_turn, action_turn, turn_number) -> TurnRecord`** in `simulation.py` (or `src/compound_turn.py` if needed).
  - Flow: execute move → rebuild view → optional look → optional turn action.
  - **Sub-steps commit independently** — no rollback of successful earlier sub-steps in the same turn.
  - Reuse Section 1 `move()`, `perform_look()`, `speak()`; interact in Section 3.

- [x] Define `TurnRecord` / history shape for compound turns

  **Agreed Decision for V0.2:**

  One record per agent turn with structured steps (V0.2.5 ingestion hook):

  ```python
  @dataclass
  class TurnStep:
      kind: Literal["move", "look", "speak", "interact"]
      reasoning: str
      target: Optional[str]
      content: Optional[str]
      result: str

  @dataclass
  class TurnRecord:
      turn_number: int
      steps: list[TurnStep]
      result: str              # composite text for prompt "Recent history"
      nav_reasoning: str
      action_reasoning: str
  ```

  - **Last 10 turns** = last 10 agent turns (one `TurnRecord` each), not sub-steps.
  - Legacy single-action `TurnRecord` fields (`action`, `target`, …) **removed** in V0.2 (breaking change; tests updated).

- [x] `passive_result` and passive vision for other agents

  **Agreed Decision for V0.2:**

  **Single `passive_result` (V0.1 model)** — passive-result **history** deferred to **V0.2.5**.

  **Which sub-step sets `passive_result`:** last **successful observable** sub-step by priority **turn action > look > move** (e.g. move + look + speak → speak line wins).

  **Confidence/emotion suffix:** from **action phase only**, appended to final `passive_result`.

  **What other agents see in passive vision** (unchanged line shape; snapshot at end of actor's compound turn):

  ```text
  Goblin (agent_goblin_01), (2, 3) - {pdesc/desc fragment} {passive_result}
  ```

  | Piece | Source |
  |-------|--------|
  | **Coordinates** | Actor's **final** position after move sub-step |
  | **Description fragment** | Observer's own look memory (`[?]` rules) |
  | **`passive_result`** | **One** line — winning sub-step only, not full compound trace |

  Observers do **not** see move + look + speak separately in V0.2 passive vision. Mid-compound state is not visible to other agents until the actor's turn completes. Richer social history → V0.2.5 memory store.

- [x] Prompt structure for two-phase turns

  **Agreed Decision for V0.2:**

  **Navigation prompt:** character + personality + room + pre-turn passive vision + move rules (Section 1) + recent history (composite) + `AgentNavigationTurn` output format. Omit speak, interact, look targets.

  **Action prompt:** character + personality + room + **post-move** passive vision + look rules + available look targets + interact list (Section 3) + speak reminder + recent history + `AgentActionTurn` output format. Omit move rules.

  **Few-shots:** `FEW_SHOT_NAV_EXAMPLES` and `FEW_SHOT_ACTION_EXAMPLES`; `fewshots` toggles both.

- [x] Logging and `session_turn` semantics

  **Agreed Decision for V0.2:**

  - **`session_turn` increments once** after `step_agent_turn` completes (orchestration finished).
  - **No increment** if navigation LLM fails before orchestration starts.
  - **Logging:** one agent turn, two sub-entries: `Turn N [nav]` and `Turn N [action]`.
  - `_run_llm_turn_for_agent`: nav LLM → move → action LLM → `step_agent_turn` → log both calls.

- [x] Manual stepper parity for compound turns

  **Agreed Decision for V0.2:**

  - **Remove** `step move|look|speak` (superseded by compound commands).
  - **Primary:** `step-compound` — one manual compound turn without LLM, e.g.:
    - `step-compound 2,3 look obj_ball_01 speak Hello.`
    - `step-compound - look obj_ball_01` (`-` or `stay` = no move)
    - `step-compound 2,3` (move only)
  - **Optional debug:** `step-nav` / `step-action` for isolated sub-phase testing (document as advanced).
  - `run` / typing agent name still runs full two-LLM compound path.

- [x] Success criteria for Section 2

  **Agreed Success Criteria for V0.2 (Section 2):**

  | # | Test | Expected result |
  |---|------|-----------------|
  | 1 | LLM `run`: nav `null`, action speak only | No move; speak runs; one `turn_number` |
  | 2 | LLM `run`: nav move + action look + speak | Steps in order; post-move look visibility |
  | 3 | Move to tile where target visible; action look | Look succeeds; would fail if prompt were pre-move |
  | 4 | Two LLM calls per `run` | `[nav]` + `[action]` logged under same agent turn |
  | 5 | Invalid look after valid move | Move committed; look error in composite `result` |
  | 6 | `turn_number` / `session_turn` | Each increments once per compound turn |
  | 7 | `step-compound 2,3 look obj_ball_01 speak Hi` | Same semantics as LLM path without API |
  | 8 | `TurnRecord.steps` | Structured sub-steps present for V0.2.5 |
  | 9 | Move + look + speak; Explorer `vision` | Goblin at new position; `passive_result` shows **speak** only |

## Section 3: Custom object actions (declarative interact)

**Depends on:** Section 2 action phase (turn action slot). Related to LONG_TERM_GOALS "objects with behaviors."

- [x] Agree philosophy: declarative effects, not freeform commands

  **Agreed Decision for V0.2:**

  - Object behaviors are **data-defined** `ObjectAction` records executed via a central **effect registry** (`src/object_effects.py` or equivalent).
  - **Not** freeform stepper command strings from object definitions (no scripting shell).
  - V0.2 ships **mutating registry effects** (`delete_self`, `random_move_self`); registry is **general** so new effects are additive handlers + docs, not interact rewrites.
  - **`effects: []` allowed** — result-only interacts (templates only, no world mutation) for flavor actions without new effect types.

- [x] Define object action data model

  **Agreed Decision for V0.2:**

  ```python
  @dataclass
  class ObjectAction:
      name: str
      range: int             # Chebyshev distance; 0 = same tile as object
      result: str            # first-person; supports {actor}, {object}
      passive_result: str    # third-person; supports {actor}, {object}
      effects: list[str]     # registry keys, ordered; may be empty
  ```

  - Attach to `Object` as **`actions: dict[str, ObjectAction]`** keyed by action name (unique per object).
  - **Multiple actions per object allowed** (e.g. future `smell` + `eat`).
  - Unknown effect names **rejected at world-edit parse time** (validated against registry).

- [x] Define range metric

  **Agreed Decision for V0.2:**

  - **Chebyshev distance:** `max(|dx|, |dy|) ≤ range` (diagonals count as distance 1).
  - **`range: 0`** — agent tile must equal object tile.
  - Measured from **post-move** agent position vs `object.position`.
  - No line-of-sight or blocker checks in V0.2.

- [x] Define V0.2 effect registry (minimum set)

  **Agreed Decision for V0.2:**

  | Registry key | Behavior |
  |--------------|----------|
  | `delete_self` | Remove the interacted object from the world |
  | `random_move_self` | Move the object to a different random in-bounds grid tile |

  - Initial world: ceramic ball includes **`kick`** (`range 1`, `random_move_self`).

  - Effects run **in list order**.
  - **`effects: []`** — no registry calls; `result` / `passive_result` templates only.
  - New effects added in later versions by extending the registry + checklist (e.g. `set_desc`, spawn).
  - **Discoverability:** read-only stepper command **`effects`** lists every registered effect name and a short description (same role as `objects` / `agents`). Referenced in `help create-object`, `help edit-object`, and stepper intro.

- [x] Define interact execution and results

  **Agreed Decision for V0.2:**

  **Trigger:** `AgentActionTurn` with `turn_action == "interact"`, `target` = object id, `action_name` = e.g. `"eat"`.

  **Validation (runtime, action phase):**
  1. Object exists.
  2. Object defines named action.
  3. Object appears in actor's **post-move passive vision** (same visibility bar as `look` on objects).
  4. Actor within **Chebyshev range** for that action.

  **On success:** run registered effects in order → substitute templates → `TurnStep` kind `"interact"` → `passive_result` wins turn-action slot (Section 2; mood suffix from action phase).

  **On failure** (composite `result`, no world change unless earlier steps committed):

  | Case | Code |
  |------|------|
  | Unknown object / action name | `ERR:UNKNOWN_INTERACT` |
  | Not in passive vision | `ERR:INTERACT_NOT_VISIBLE` |
  | Out of range | `ERR:INTERACT_OUT_OF_RANGE` |

  **Cookie example:** `delete_self`; `result` `You ate the cookie, it was delicious.`; `passive_result` `{actor} ate the cookie.`

- [x] Available actions in action-phase prompt

  **Agreed Decision for V0.2:**

  New block in **action prompt only** (post-move):

  ```text
  Object interactions available this turn:
  - eat obj_cookie_01 (Cookie) — range 1
  ```

  - List actions where object is **in passive vision** and **in range**.
  - Omit section when empty.
  - Format: `- {action_name} {object_id} ({display name}) — range {n}` (or `same tile` when `range: 0`).

- [x] World edit commands for object actions

  **Agreed Decision for V0.2:**

  **`create-object`** — optional action fields:

  ```text
  create-object name "Cookie" pdesc "A cookie." desc "A tasty cookie." at 2,2 \
    action eat range 1 effect delete_self \
    result "You ate the cookie, it was delicious." \
    passive "{actor} ate the cookie."
  ```

  - `effect` is optional; omit for `effects: []`. Single `effect` keyword in V0.2 maps to one-element `effects` list (repeatable `effect` tokens deferred).

  **`edit-object`:**
  - `edit-object obj_cookie_01 add-action eat range 1 effect delete_self result "..." passive "..."`
  - `edit-object obj_cookie_01 remove-action eat`

  - Parse-time validation: known registry effects, non-negative `range`.
  - **`objects` / `list`:** show attached action names per object (e.g. `actions: eat`).

- [x] Interaction with look / move / speak

  **Agreed Decision for V0.2:**

  - **Interact occupies turn-action slot** — no `speak` same action phase (Section 2).
  - Runs after optional `look_target` in action phase.
  - **Manual:** `step-compound … interact obj_cookie_01 eat` (with nav/look variants as needed).
  - **`delete_self` cleanup:** on any object removal, purge that `object_id` from all agents' `looked_at` and `ever_looked` (shared `remove_object` path — applies to `delete-object` and interact).

- [x] Success criteria for Section 3

  **Agreed Success Criteria for V0.2 (Section 3):**

## Section 4: Cross-cutting concerns (integration)

- [x] Backward compatibility with V0.1

  **Agreed Decision for V0.2:**

  **Accepted breaking changes:**

  | Area | V0.1 | V0.2 |
  |------|------|------|
  | Move | Cardinal one-step | Coordinate teleport (`"x,y"`) |
  | Agent turn | Single `AgentTurn` / one LLM call | `AgentNavigationTurn` + `AgentActionTurn` / two LLM calls |
  | `TurnRecord` | Flat `action` / `target` / `content` | `steps[]` + `nav_reasoning` / `action_reasoning` + composite `result` |
  | Manual step | `step move\|look\|speak` | `step-compound` (+ optional `step-nav` / `step-action`) |

  **Explicitly preserved:** 5×5 grid, world editing, multi-agent passive vision, `looked_at` / `ever_looked`, single `passive_result`, ManualStepper + logging, `switch` / `run` / typing agent name.

  **`AgentTurn`:** **Removed** from the LLM path (replaced by navigation + action schemas). Update `docs/schemas/` reference copies; migrate tests.

  **No compatibility shim** for cardinal moves or single-action `TurnRecord` in V0.2 code paths.

- [x] Error codes and failure strings

  **Agreed Decision for V0.2:**

  Keep `ERR:{NAME}` in `result` strings and validation messages.

  | Code | When |
  |------|------|
  | `INVALID_JSON` | LLM output fails Pydantic parse (nav or action) |
  | `INVALID_TARGET` | Malformed or wrong-type target (e.g. bad look id) |
  | `INVALID_COORDINATES` | Move / `move_target` unparseable or out of bounds |
  | `INTERACT_NOT_VISIBLE` | Interact target not in post-move passive vision |
  | `INTERACT_OUT_OF_RANGE` | Interact valid but beyond action `range` |
  | `UNKNOWN_INTERACT` | Unknown object id or action name on object |
  | `CONTENT_TOO_LONG` | Speak limits exceeded |
  | `REASONING_TOO_LONG` | Reasoning > 400 chars |
  | `INVALID_CONTENT` | `confidence` / `emotion` too long |

  **LLM abort before orchestration** (nav/action parse failure): `log_error` only — no `TurnRecord`, no `session_turn` increment. Sub-step failures inside a completed compound turn appear in composite `result` with codes above.

  **Speak character limit:** **500 characters** (5 sentences unchanged). Replaces V0.1 280-char cap.

- [x] Token budget and prompt size

  **Agreed Decision for V0.2:**

  - **No hard prompt truncation** in V0.2.
  - **Soft monitoring:** log prompt length per `[nav]` / `[action]` call.
  - **Informal target:** each phase < ~4k characters on the initial 5×5 room; revisit if testing exceeds this.
  - **Truncation policy:** deferred to V0.2.5 prompt assembler.

- [x] Observability (logging, stepper, help)

  **Agreed Decision for V0.2:**

  **Logging:** `Turn N [nav]` and `Turn N [action]` under one agent turn / `session_turn`; log prompts, raw outputs, parsed schemas, token usage per sub-call.

  **`state`:** show active agent's last `TurnRecord` with **step breakdown** (`kind`, `target`, `result`); `passive_result`, position, turn count.

  **Intro / help (shipped):** coordinate move, two LLM calls on `run`, `step-compound`, `effects`, removal of `step move|look|speak`. `help` for `step-compound`, `step-nav`, `step-action`, `effects`. Run via `uv run realm` (after `uv sync`) or `uv run python src/main.py`.
