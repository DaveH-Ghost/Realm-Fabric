## Why 0.6.1

V0.2 introduced a fixed effect registry (`delete_self`, `random_move_self`, `move_area`) in `src/object_effects.py`. V0.6.0 added **triggers** that reuse the same registry. That works for the demo but blocks richer app behavior (eat food, lock doors, spawn items, cutscene scripts) without forking the engine.

**0.6.1** moves behavior to **registered handlers** — same sustainability model as pluggable memory modules (V0.2.5). The engine keeps:

- `ObjectAction` templates (`result`, `passive_result`)
- Range, kind (`interact` | `trigger`), trigger flags
- Calling the handler after templates (interact) or after area event (trigger)

Apps (and realm-studio as a reference app) own handler implementations.

## Design summary

### `InteractionHandler` protocol

```python
# src/interaction_handlers/base.py (proposed)

class InteractionHandler(Protocol):
  def __call__(
      self,
      session: Session,
      agent: Agent,
      obj: Object,
      action: ObjectAction,
  ) -> str | None:
      """Return None on success, or an in-world error string."""
      ...
```

Optional companion hooks (mirror memory modules):

- **`validate_params(params: dict[str, str]) -> str | None`** — validate before save / CLI parse
- **`describe() -> str`** — human label for `handlers` listing and studio picker

### Registry API

```python
register_interaction_handler(
    handler_id: str,
    handler: InteractionHandler,
    *,
    description: str = "",
    validate_params: Callable[[dict[str, str]], str | None] | None = None,
)
list_registered_handlers() -> list[str]
get_handler(handler_id: str) -> InteractionHandler | None
is_handler_registered(handler_id: str) -> bool
```

Process-wide registry (like `memory_modules.registry`). **No built-in handlers in the engine** after 0.6.1c.

### `ObjectAction` model change

| Field | 0.6.0 | 0.6.1 |
|-------|-------|-------|
| `effects: list[EffectSpec]` | ✅ | **Removed** |
| `handler_id: str \| None` | — | Optional; `None` = flavor-only (templates only) |
| `handler_params: dict[str, str]` | — | Opaque key/value strings for the handler |

**One handler per action** (not an effect chain). Apps that need multi-step behavior implement one handler that does several mutations.

Trigger-only engine flags **unchanged**: `halt_movement`, `delete_after_trigger`, `trigger_exceptions`.

**Trigger execution order (unchanged semantics, clearer contract):**

1. Format `passive_result` → `Session.emit_area_event` (if non-empty)
2. If `handler_id` set → run handler (errors surface to interact path; triggers log / skip delete on error — define in 0.6.1c)
3. Else if `kind == "trigger"` and `delete_after_trigger` → remove object

### What leaves the engine

- `src/object_effects.py` — deleted or reduced to a thin shim during migration only
- `src/effect_spec.py` — removed (replaced by `handler_params`)
- Built-in effect handlers (`_delete_self`, `_random_move_self`, `_move_area`) — move to **reference handlers** package

### What stays in the engine

- `interact.py` — pathing, range, template formatting, then `run_handler(...)`
- `triggers.py` — path-step evaluation, area events, then `run_handler(...)`
- `interact_templates.py` — `{actor}`, `{object_end}`, area ids, etc.

## Save format (`snapshot_version: 4`)

New saves use **`snapshot_version: 4`**. Import of **v1–v3** remains supported.

### `actions_detail[]` serialization

```json
{
  "name": "kick",
  "range": 0,
  "result": "You kick the ball.",
  "passive": "{actor} kicks the ball.",
  "kind": "interact",
  "handler_id": "random_move_self",
  "handler_params": {}
}
```

v3 import migration for each action:

| v3 `effects` | v4 mapping |
|--------------|------------|
| `[]` | `handler_id: null`, `handler_params: {}` |
| `[{ "name": "delete_self" }]` | `handler_id: "delete_self"` |
| `[{ "name": "random_move_self" }]` | `handler_id: "random_move_self"` |
| `[{ "name": "move_area", "params": { ... } }]` | `handler_id: "move_area"`, copy params |
| Multiple effects | Import **warning**; use first effect only (document; rare in saves) |

On import, if `handler_id` is non-null and **not registered** in the current process → **fail import** with clear message (same pattern as missing custom memory modules).

### Session import checklist (extended)

1. Custom memory modules loaded
2. **Interaction handlers registered** for every `handler_id` referenced in the save

  - `random_move_self.py`

- **Default profile** ball `kick` action uses `handler_id: random_move_self` in profile JSON / area seed (if actions are stored there) or remains created via area init in `area.py`

**CLI**

```
create-object ... handler random_move_self
create-object ... handler move_area dest-area hall dest-at 0,0
edit-object <id> add-action enter range 0 handler move_area dest-area hall dest-at 0,0 ...
handlers   # replaces effects
```

- Remove `effect` keyword (breaking; document in changelog)
- `dest-area` / `dest-at` only valid with `handler move_area`

**Tests:** stepper parse, handlers command lists reference set, demo ball kick still works

## Reference handlers (realm-studio + CLI)

| `handler_id` | Params | Behavior |
|--------------|--------|----------|
| `delete_self` | — | Remove object from area |
| `random_move_self` | — | Move object to random in-bounds empty tile |
| `move_area` | `dest-area`, `dest-at` | `Session.transfer_agent` for interacting agent |

Apps add handlers the same way: register at process start before session import or agent turns.

**Example app pattern**

```python
from realm_fabric import Session, register_interaction_handler

def eat_food(session, agent, obj, action):
    session.remove_object(obj.id)
    return None

register_interaction_handler("eat_food", eat_food, description="Remove object (eaten)")

session = Session.from_snapshot(...)
```

## Breaking changes (0.6.0 → 0.6.1)

| Area | Change |
|------|--------|
| CLI | `effect` → `handler`; `effects` command → `handlers` |
| Snapshots | New saves are v4; v3 imports migrate `effects[0]` → handler |
| API | `actions_detail[].effects` removed; use `handler_id` + `handler_params` |
| Engine | No built-in effects; must register handlers before import / interact |
| Multi-effect actions | Not supported; combine in one handler |
