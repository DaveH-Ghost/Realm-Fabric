# Version 0.3.0 Changelog

**Purpose:**  
Living record of what ships in **V0.3.0** and why — **not** a pre-implementation checklist. Each section corresponds to an incremental release slice (e.g. `0.3.0a`, `0.3.0b`) so we can implement and review agilely without a full spec up front.

**Baseline:** Builds on **V0.2.5** (`v0.2.5`) — single compound LLM call, pluggable memory modules, Passive Vision–first prompt, compound turn execution (move → look → turn action).

**Overall V0.3.0 focus:** Turn Realm-Fabric into a **reusable engine** — a pip-installable library with a stable session API, serializable world state, and pluggable game profiles (prompt templates, default worlds). The CLI becomes a thin reference client. **No web UI in 0.3.0** — that ships in **V0.3.1** as an example project built on this engine.

See also: [ROADMAP.md](../ROADMAP.md), [v0.2.5-changelog.md](v0.2.5-changelog.md), [LONG_TERM_GOALS.md](../../LONG_TERM_GOALS.md).

## Why V0.3.0 before web

Realm-Fabric should work as an **engine** others can build on (D&D NPC tools, slice-of-life RP, strategy games, etc.). A web GUI is the first *consumer*, not the core.

Shipping the engine first avoids:

- Web-specific APIs baked into `main.py` that never migrate into the library
- Duplicate world state (Python sim vs frontend)
- Prompt logic trapped in one monolithic `build_compound_prompt()` that every game must fork

**V0.3.0** proves the library contract with pytest + CLI. **V0.3.1** proves it with a browser.

## 0.3.0a — Session API

**Status:** ✅ **Implemented** in code. CLI refactor to use `Session` → **0.3.0d** ✅.

**Goal:** One object owns runtime state and exposes the operations every client needs (CLI today, HTTP handlers in 0.3.1).

### Shipped API (`src/session.py`)

| Type / method | Detail |
|---------------|--------|
| **`Session.from_default()`** | Demo world (Explorer, ball, sign) |
| **`Session(world, active_agent_id=…)`** | Custom world + active agent |
| **`get_active_agent()` / `get_agent(name_or_id)`** | Resolve by id or case-insensitive name |
| **`set_active_agent(name_or_id)`** | → `SessionResult` |
| **`run_compound_turn(turn, agent_id=…)`** | → `TurnResult` with `TurnRecord`; increments `session_turn` |
| **`build_prompt(name_or_id=…)`** | Compound prompt via session **`GameProfile`** |
| **`Session.from_profile(profile)`** | Session from profile's default area factory |
| **`run_command(line)`** | World edits + listings → `CommandResult` |

### Command coverage

`create-object`, `edit-object`, `delete-object`, `create-agent`, `edit-agent`, `delete-agent`, `list`, `objects`, `agents`, `effects`, `memory-modules`.

Not in `run_command` (use dedicated methods): `step-compound`, LLM turns (stay in CLI until **0.3.0d** / later `run_llm_turn`).

### Tests

- `tests/test_session.py` — **13 tests** including web-handler-style flow (create → move → interact)
- Full suite green at implementation time

### Explicitly deferred to later slices

- `snapshot()` JSON — **0.3.0b** ✅
- `GameProfile` template files — **0.3.0c** ✅ (`PromptContext` shipped in 0.3.0a)
- `ManualStepper` → `Session` — **0.3.0d** ✅

### Addendum — configurable world + `PromptContext`

- **`Area`**: `GridBounds` + `area_description` at construction; `create_area(width=, height=, …)`
- **`build_prompt_context(agent, area)`** → `PromptContext` (`passive_vision`, `memory`, `area_description`, …) for application-layer templates
- Default `build_compound_prompt` = `assemble_default_compound_prompt(ctx)` (reference only)
- Tests: `tests/test_area_config.py`

### Rename — `World` → `Area` (pre-0.3.0b)

- `World` / `world.py` → **`Area`** / **`area.py`**; `world_edit` → **`area_edit`**
- `room_description` → **`area_description`**; `Session.area` holds the active grid (one area per session until **V0.4**)

## 0.3.0b — Area snapshot / serialization

**Status:** ✅ **Implemented** in code.

**Goal:** Clients (web, save/load later) consume **JSON-friendly state**, not dataclass internals.

### Shipped API

| Piece | Detail |
|-------|--------|
| **`Session.snapshot()`** | Dict with grid, `area_description`, `session_turn`, `active_agent_id`, agents, objects |
| **`build_area_snapshot(area, …)`** | Same shape without a `Session` (testing / future multi-area) |
| **`serialize_agent` / `serialize_object`** | Public field helpers |
| **`include_private=False`** (default) | Omits `personality`, `passive_description`, `description` on agents |
| **`include_passive_vision=True`** (default) | Adds engine-built `passive_vision` for active agent |

### Snapshot shape

```json
{
  "grid": { "min_x": 0, "max_x": 4, "min_y": 0, "max_y": 4 },
  "area_description": "...",
  "session_turn": 0,
  "active_agent_id": "agent_01",
  "passive_vision": "You are at (1, 1).\n...",
  "agents": [
    {
      "id": "agent_01",
      "name": "Explorer",
      "position": [1, 1],
      "passive_result": "",
      "memory_module": "recent_turns"
    }
  ],
  "objects": [
    {
      "id": "obj_ball_01",
      "name": "Ceramic Ball",
      "position": [2, 2],
      "actions": ["kick"]
    }
  ]
}
```

### Explicitly not in 0.3.0b

- Full round-trip `load_snapshot()` → `Area` (optional later)
- Memory turn history in snapshot (module id only)
- Encrypted saves, multiplayer sync

### Tests

- `tests/test_session_snapshot.py` — shape, JSON serializable, privacy, post-turn updates, passive vision
- Full suite green at implementation time

## 0.3.0c — GameProfile + prompt templates

**Status:** ✅ **Implemented** in code.

**Goal:** Projects customize **prompt prose and scenario defaults** without forking core sim code — similar in spirit to SillyTavern prompt editing, but structured.

### Shipped API

| Piece | Detail |
|-------|--------|
| **`GameProfile`** (`src/game_profile.py`) | `profile_id`, `schema_id`, template, few-shots, `create_area` factory |
| **`PromptTemplate`** (`src/prompt_template.py`) | `{{slot}}` substitution from `PromptContext` fields (`rules` aliases `compound_rules`) |
| **`default_compound_profile()`** | Loads `profiles/default_compound/` — same compound prompt behavior as pre-0.3.0c |
| **`Session(profile=…)`** / **`from_profile()`** | Prompt assembly via `profile.build_prompt(ctx)` |
| **`profile.turn_schema()`** | Returns `AgentCompoundTurn` for `schema_id="AgentCompoundTurn"` (swappable in **V0.4**) |

### Reference profile (`profiles/default_compound/`)

| File | Role |
|------|------|
| `template.txt` | Default section order: Passive Vision, rules, memory, output format |
| `few_shots.txt` | Compound turn examples (appended when `include_examples=True`) |

`build_compound_prompt()` / `assemble_default_compound_prompt()` delegate to the default profile for backward compatibility.

### Design rules (unchanged)

- Core **always** builds `PromptContext` from sim state.
- Profile **never** computes game rules — only text layout and instructions.
- **`AgentCompoundTurn`** remains the only turn schema until **V0.4**.

### Explicitly not in 0.3.0c

- GUI prompt editor
- Swappable turn schemas / executors — **V0.4**
- Profile marketplace / hot-reload in production

### Tests

- `tests/test_game_profile.py` — default sections, parity with `build_compound_prompt`, custom minimal template, area factory equivalence
- Full suite green at implementation time (266 tests)

## 0.3.0d — CLI refactor

**Status:** ✅ **Implemented** in code.

**Goal:** `ManualStepper` is a **reference client** of `Session`, not the owner of simulation state.

### Shipped changes

| Piece | Detail |
|-------|--------|
| **`ManualStepper.session`** | Created via `Session.from_profile(default_compound_profile())` |
| **Turns / commands** | `run`, agent-name default, `step-compound`, world-edit, listings → `session.*` |
| **`vision` / `prompt` / `state`** | Passive vision + `session.build_prompt()` + `session.format_debug_state()` |
| **Backward compat** | `stepper.area`, `stepper.agent`, `stepper.session_turn`, `stepper.include_examples` properties |
| **`Session.format_debug_state()`** | CLI `state` report moved to engine |
| **`Session.gate_agent_turn()`** | Pre-LLM consolidation gate (public) |

### Unchanged in spirit

- `step-nav` / `step-action` — debug-only; still call simulation phases directly (no `TurnRecord`)
- File logging via `log_turn` / `log_error` in `main.py` (I/O layer)

### Tests

- Existing `test_stepper.py`, `test_multi_agent.py`, `test_area_edit.py` green
- `test_stepper_delegates_to_session`; `test_session.py` — `format_debug_state`, `gate_agent_turn`
- Full suite green at implementation time (269 tests)
