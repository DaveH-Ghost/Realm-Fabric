# Version 0.1 Implementation Readiness Checklist

> **As shipped in v0.1.0.** **V0.2 on `main` supersedes** cardinal move, single-action `AgentTurn`, flat `TurnRecord`, and `step move|look|speak` — see [v0.2-implementation-readiness-checklist.md](v0.2-implementation-readiness-checklist.md) and [ROADMAP.md](ROADMAP.md). Tag `v0.2.0` pending.

**Purpose:**  
This checklist defines the key design decisions, scope, and agreements for V0.1. It builds on the validated V0 foundation (reliable single-action-per-turn structured output with full observability). **As shipped in v0.1.0**, it covers four areas: stale perception, world editing, multi-agent observation, and observable actions.

**Rule:** No code changes for V0.1 features until the relevant items on this checklist are checked off. (Use `[doc]` for checklist updates; `[code]` only when explicitly authorized for implementation.)

**Project Scope (Current Agreement — as shipped in v0.1.0):**
- Builds directly on V0: same 5x5 grid, 3 actions (move/look/speak), one action per LLM call/turn, TurnRecord/AgentTurn models, rich logging + ManualStepper for observability.
- Multiple agents share the same world with **independent memory** and **mutual passive observation** (other agents visible in vision; you do not see yourself).
- Runtime creation/editing/deletion of objects and agents via stepper commands (replaces the V0 special-case `sign` command).
- "Has changed" notifications are generalized to **any object or agent `desc`** and apply across *all* agents that have looked at that entity.
- Default initial world (`create_initial_world`): same layout as V0 (Explorer + ball + sign) but Explorer uses **`pdesc` / `desc` / `personality`** (personality text moved off the old single `description` field). New agents/objects are added at runtime via commands.
- Fully manual control: human decides which agent gets a turn by typing its name, using `run` (LLM turn for the active agent), or using `switch` + `step` / `run`.
- Goal: Enable richer experimentation with dynamic environments and multiple agents while preserving full prompt→reasoning→action→memory/perception loops.
- **Out of scope for V0.1:** speak targeting, relationships/beliefs, agent-initiated world edits, automatic turn sequencing, compound actions per turn, any GUI/visuals (see ROADMAP.md for V0.3), changes to the one-action-per-turn model, new dependencies.
- **Speak validation (shipped):** up to **5 sentences** and 280 characters; pure dialogue encouraged via prompt only (no runtime emote/parentheses rejection).

See also: [ROADMAP.md](ROADMAP.md) for high-level V0.1 summary and [LONG_TERM_GOALS.md](../LONG_TERM_GOALS.md) for longer-term aspirations.

## Section 1: Generalize the "has changed" notification

**Implementation status:** ✅ Implemented. Stale wording and initial-world sign behavior were refined by the **perception extension** (Section 2, below): passive (`pdesc`) + detailed (`desc`) descriptions; stale text is `[?] [changed] {passive}`.

- [x] Define the generalized notification text format

  **Agreed Decision for V0.1 (as shipped):**

  Passive vision states are rendered via `format_object_vision_desc()` (see perception extension for full `pdesc`/`desc` matrix). Core cases:

  | State | Text after `-` (typical) |
  |-------|----------------|
  | **Current knowledge** | Detailed `obj.description` (or passive if no detailed) |
  | **Never examined** (detailed hidden) | `[?]` or `[?] {passive}` |
  | **Stale** (examined before, detailed invalidated) | `[?] [changed]` or `[?] [changed] {passive}` |

  **Format rules:**
  - Line shape: `{name} ({id}), {position} - {fragment}` (omit ` - ` when fragment is empty).
  - `[?]` only when a **detailed** description exists and is hidden or stale.
  - Neutral, object-agnostic — no sign-specific phrasing.

  **Examples (shipped):**

  ```
  Ceramic Ball (obj_ball_01), (2, 2) - [?]
  Ceramic Ball (obj_ball_01), (2, 2) - [?] [changed]
  Ceramic Ball (obj_ball_01), (2, 2) - A slightly worn ceramic ball. It has a few scuffs and feels light.
  Wooden Sign (obj_sign_01), (2, 4) - [?] A simple wooden sign on the wall.
  ```

- [x] Decide on tracking mechanism for stale/known state (per agent)

  **Agreed Decision for V0.1:**

  Add a second per-agent set, `ever_looked`, to `Memory`. `looked_at` alone cannot distinguish "never seen" from "seen but now stale."

  | Set | Meaning | Updated when |
  |-----|---------|--------------|
  | `looked_at` | Agent has **up-to-date** knowledge of this object | Added by `mark_looked_at`; removed by `invalidate_look` |
  | `ever_looked` | Agent has looked at this object **at least once** | Added by `mark_looked_at`; cleared by `clear_examination` when detailed removed |

  **API changes on `Memory`:**
  - `mark_looked_at(object_id)` → adds to both `_looked_at` and `_ever_looked`
  - `invalidate_look(object_id)` → removes from `_looked_at` only; `_ever_looked` unchanged
  - `clear_examination(object_id)` → removes from both sets (e.g. when `desc` cleared)
  - `has_ever_looked_at(object_id) -> bool`
  - Existing `has_looked_at`, `looked_at` property, and `invalidate_look` remain the primary mutation APIs

  **We are not adding a separate `stale` set.** Stale state is derived: `not has_looked_at(id)` and `has_ever_looked_at(id)` and detailed description still exists.

  **Initial world behavior (as shipped):**
  - No pre-mark on sign; agent memory starts empty.
  - Ball: detailed only → `[?]`. Sign: passive + detailed → `[?] A simple wooden sign on the wall.`

  **Edge cases:**
  - New objects created at runtime (Section 2): not in either set → vision per `pdesc`/`desc` rules.
  - Deleted objects: no memory cleanup required; object simply no longer appears in vision.
  - `reset_looked_at()` clears both `looked_at` and `ever_looked`.

- [x] Update passive vision logic

  **Agreed Decision for V0.1:**

  Implemented in `format_object_vision_desc()` (see perception extension for `pdesc`/`desc` rules). Stale branch uses `[?] [changed]` (+ passive when set); only when detailed text still exists.

  **Unchanged from V0:**
  - Header line: `You are at (x, y).`
  - All objects in the world appear in passive vision (no range/occlusion in V0.1).
  - `perform_look` remains valid on any object visible in passive vision, including `[?]` and changed states.
  - `get_visible_object_ids` and `get_available_look_targets` behavior unchanged.

  **Tests (implemented in `tests/test_perception.py`):**
  - Ball: `[?]` → detailed after look → `[?] [changed]` after invalidation → detailed after re-look.
  - Sign: `[?] {passive}` at start → stale `[?] [changed] {passive}` after examined + `desc` edit.

- [x] Define invalidation on edit (cross-agent)

  **Agreed Decision for V0.1:**

  Add a world-level helper on `World`:

  ```python
  def invalidate_entity_knowledge(self, entity_id: str) -> None:
      """Remove up-to-date look knowledge for this object or agent across all agents."""
      for agent in self.agents:
          if agent.memory.has_looked_at(entity_id):
              agent.memory.invalidate_look(entity_id)
  ```

  (`invalidate_object_knowledge` remains as an alias for objects.)

  **When to call it (V0.1):**
  - Object or agent **`desc`** changes → always call `invalidate_entity_knowledge`
  - Object **name** or **position** changes only → **do not** invalidate (description unchanged; agent's textual knowledge is still accurate)

  **Who calls it:**

  | Phase | Caller |
  |-------|--------|
  | Section 2 (shipped) | `edit-object` calls it when `desc` changes to a non-empty value |
  | Section 2 (shipped) | `edit-object` with empty `desc` calls `clear_object_examination_history` instead |

  **Cross-agent rules:**
  - Only agents with `has_looked_at(id)` are invalidated on `desc` change.
  - An agent who never looked continues to see never-examined vision (`[?]` or `[?] {passive}`).
  - `perform_look` calls `mark_looked_at` when detailed text exists.

- [x] Update related areas

  **Agreed Decision for V0.1:**

  **Prompt / few-shots (`prompt.py`):** ✅ Updated — Example 4 uses `[?] [changed]`; system rules describe passive, `[?]`, and stale states.

  **`sign` command:** Removed in Section 2; use `edit-object obj_sign_01 desc "..."`.

  **`perform_look`:** Returns detailed text; no-detail message (no memory update) when `desc` is empty.

  **`state` command (optional enhancement):**
  - May show both `looked_at` and `ever_looked` sets for debugging (e.g. `Looked at (current): [...]` and `Ever looked at: [...]`). Not required for Section 1 success, but recommended during implementation and testing.

  **Logging:**
  - No structural changes. Updated vision text will appear automatically via existing `vision` / `prompt` / turn logging.

## Section 2: General editing commands (objects + agents)

- [x] Replace the special `sign` command

  **Agreed Decision for V0.1:**

  Remove `do_sign` entirely (including hard-coded `obj_sign_01` and single-agent invalidation). Replace with keyword-style stepper commands, parsed via `shlex.split` for quoted strings.

  **Object commands:**

  ```
  create-object name "Ceramic Ball" desc "A worn ball." at 2,2
  edit-object obj_ball_01 desc "New description here."
  edit-object obj_ball_01 name "Old Ball"
  edit-object obj_ball_01 pos 3,3
  edit-object obj_ball_01 desc "..." name "..." pos 2,2    # multiple fields, any order
  delete-object obj_ball_01
  ```

  **Agent commands:**

  ```
  create-agent name "Goblin" pdesc "A short figure." desc "A grumpy goblin." personality "You are a grumpy goblin." at 0,3
  edit-agent agent_01 desc "Updated appearance."
  edit-agent agent_01 personality "Updated LLM personality."
  edit-agent agent_01 name "Scout" pos 2,1
  delete-agent agent_02
  ```

  **Listing commands (read-only, no turn consumed):**

  ```
  objects     # list all objects
  agents      # list all agents
  list        # list everything (agents + objects)
  ```

  **Syntax rules:**
  - Keywords (`name`, `desc`, `at`, `pos`) are case-insensitive.
  - Quotes are required for `name` and `desc` values that contain spaces.
  - `at` / `pos` use `x,y` integer format with no spaces (e.g. `2,2`).
  - `edit-*` and `delete-*` commands take the entity **id** as the first positional argument (not display name). Use `objects` / `agents` / `list` to look up ids.
  - `edit-*` commands require at least one field to change.
  - All create/edit/delete commands and all listing commands (`objects`, `agents`, `list`) are **world-level** and do **not** consume a turn.

  **`objects` — list all objects:**
  - Lists **every** object currently in the world with **id**, **name**, and **position**.
  - Example output:
    ```
    Objects in world:
      - Ceramic Ball (obj_ball_01) at (2, 2)
      - Wooden Sign (obj_sign_01) at (2, 4)
    ```
  - If no objects exist: print a clear empty-state message.

  **`agents` — list all agents:**
  - Lists **every** agent currently in the world with **id**, **name**, **position**, and an **(active)** marker on the current active agent.
  - Enhance the existing `do_agents` handler (already present in V0) to include agent ids in the output.
  - Example output:
    ```
    Agents in world:
      - Explorer (agent_01) at (1, 1) (active)
      - Goblin (agent_goblin_01) at (0, 3)
    ```
  - If only one agent exists, still list it with id and active marker.

  **`list` — list everything:**
  - Single command that prints **both** the full agents list and the full objects list (same content and format as running `agents` then `objects`).
  - Purpose: one-shot world overview for editing sessions without running a turn or viewing a prompt.
  - Example output:
    ```
    Agents in world:
      - Explorer (agent_01) at (1, 1) (active)

    Objects in world:
      - Ceramic Ball (obj_ball_01) at (2, 2)
      - Wooden Sign (obj_sign_01) at (2, 4)
    ```
  - Does **not** include agent memory, turn count, or few-shot settings (those remain in `state`).
  - Always prints **both** sections (agents, then objects). If a section is empty, print the header plus a clear empty-state line (e.g. `No objects in world.`). With at least one agent required in the world, the agents section is never empty in normal use.

  **Migration from V0 `sign` command:**
  - Old: `sign This is new text.`
  - New: `edit-object obj_sign_01 desc "This is new text."`
  - Run `objects` or `list` to confirm the sign id (`obj_sign_01`) if unsure.

  **Typical editing session (listing commands are central to the workflow):**

  ```
  list                                                             # quick overview of entire world
  objects                                                          # copy object ids for edit/delete
  agents                                                           # copy agent ids for edit/delete
  create-object name "Crate" desc "A wooden crate." at 0,0
  objects                                                          # confirm new id (e.g. obj_crate_01)
  edit-object obj_crate_01 desc "A sturdy wooden crate."
  edit-object obj_sign_01 desc "Updated sign text."                # replaces old sign command
  list                                                             # verify agents + objects after edits
  delete-object obj_crate_01
  objects                                                          # confirm object removal
  ```

  **Documentation:**
  - Update stepper `intro`, per-command docstrings (`help create-object`, `help objects`, `help agents`, `help list`, etc.), and README with the syntax, examples, and typical session flow above.
  - The stepper `intro` must list **`list`**, **`objects`**, and **`agents`** alongside create/edit/delete commands (not buried in help only).

  **Implementation note:** Keep `ManualStepper` as the entry point; implement `do_objects`, `do_list`, and enhance `do_agents` on `ManualStepper`. Consider shared helpers (e.g. `_format_agents_list()`, `_format_objects_list()`) so `list` reuses the same output as `agents` + `objects`. Parsing and validation for mutating commands may live in a small helper module (e.g. `src/world_edit.py`) to avoid bloating `main.py`. No CLI framework (Typer/Click) for V0.1.

- [x] ID uniqueness rules

  **Agreed Decision for V0.1:**

  **Uniqueness scope:**

  | Scope | Rule |
  |-------|------|
  | Object IDs | Unique among objects (`obj_*` prefix) |
  | Agent IDs | Unique among agents (`agent_*` prefix) |
  | Cross-category | Prefixes keep namespaces separate in practice |
  | Display names (objects) | May duplicate freely (two objects can both be named `"Box"`) |
  | Display names (agents) | Must be **unique among agents** (case-insensitive), because the stepper dispatches LLM turns by typing the agent's name |
  | Agent names vs commands | Must **not** match any stepper command (case-insensitive; hyphen/underscore variants count). Validated via `src/stepper_commands.py`, which derives reserved names from `ManualStepper`'s `do_*` handlers at runtime |
  | Object ↔ agent names | May match (e.g. an object and an agent both named `"Guard"`) |

  **Auto-ID generation (create commands only):**
  1. Sanitize display name: lowercase, spaces → underscores, strip non-alphanumeric (e.g. `"Ceramic Ball"` → `ceramic_ball`).
  2. Prefix: `obj_{slug}_` or `agent_{slug}_`.
  3. Append zero-padded counter starting at `01`; increment until unused in that category.
  4. Examples: `obj_ceramic_ball_01`, `agent_goblin_01`.
  5. Fallback if slug is empty: `obj_new_01` / `agent_new_01`.

  **Immutability:**
  - IDs are assigned at creation and **never change** in V0.1. No `edit-id` command.
  - Explicit user-supplied IDs on create are out of scope for V0.1 (auto-generated only).

  **World helpers to add (during implementation):**
  - `get_agent_by_id(agent_id) -> Optional[Agent]`
  - `remove_object(object_id) -> bool`
  - `remove_agent(agent_id) -> bool`
  - ID existence checks as needed for generation (scan existing lists; no separate ID utils module).

  **Discovering IDs (important for the editing workflow):**
  - Auto-generated ids are not typed by the user on create — the create confirmation prints the new id, but listing commands are the primary way to look up ids for subsequent edits.
  - **`objects`** — look up object ids for `edit-object` / `delete-object`.
  - **`agents`** — look up agent ids for `edit-agent` / `delete-agent`.
  - **`list`** — quick overview of all ids when unsure which category an entity is in.
  - `edit-object` / `delete-object` always use object **id**, not display name (names may duplicate).
  - `edit-agent` / `delete-agent` always use agent **id**, not display name (even though agent names are unique among agents, commands are id-based for consistency).
  - After `create-object` / `create-agent`, confirmation messages should remind the user of the listing commands (e.g. `Use 'objects' or 'list' to see all object ids.` / `Use 'agents' or 'list' to see all agent ids.`).

- [x] Editing behavior and side effects

  **Agreed Decision for V0.1:**

  **Object edits:**

  | Field changed | Effect |
  |---------------|--------|
  | `desc` | Update description → call `world.invalidate_entity_knowledge(id)` (per Section 1) |
  | `name` | Update display name only — **no** invalidation |
  | `pos` | Update position — **no** invalidation; must pass `world.is_valid_position()` |

  **Agent edits:**

  | Field changed | Effect |
  |---------------|--------|
  | `desc` | Update observable detailed text → call `world.invalidate_entity_knowledge(id)` (same stale rules as objects) |
  | `pdesc` | Update glance text in other agents' passive vision — **no** invalidation |
  | `personality` | Update private LLM prompt text for that agent only — **no** invalidation, never shown in vision or look |
  | `name` | Update display name and re-key `self.agents` dict (case-insensitive); reject if new name collides with another agent or a stepper command |
  | `pos` | Move agent — must be in bounds; memory untouched |

  **Create behavior:**
  - Position must be a valid grid coordinate.
  - Multiple entities per tile are **allowed** (consistent with V0: no blocking objects).
  - Empty descriptions are allowed.
  - New objects appear as `[?]` for all agents (not in any agent's `looked_at` or `ever_looked`).
  - New agents get a fresh `Memory()` and are registered in `self.agents` dict (basic sync; Section 3 polishes multi-agent turn behavior).
  - **`create-agent` does not change the active agent** — `self.agent` stays whoever was active before (usually Explorer). To run the new agent's turn, type its name, or `switch` to it then `run`.

  **Delete behavior:**
  - `delete-object`: remove from `world.objects`; no memory cleanup required.
  - `delete-agent`: remove from `world.agents` and `self.agents` dict.
  - If the deleted agent was active (`self.agent`), fall back to `world.agents[0]` after removal and print `Active agent: …` (same format as `switch`).
  - **Reject** deleting the last agent in the world — at least one agent must remain.
  - Deleting initial entities (ball, sign, Explorer) is allowed.

  **Sign workflow:**
  - Fully replaced by `edit-object obj_sign_01 desc "..."` with the same Section 1 invalidation path as any other object description change.

  **Live updates:**
  - After any create/edit/delete, world state and the active agent's `vision` / `prompt` reflect changes immediately (no restart).
  - **`objects`, `agents`, and `list` all reflect changes immediately** — create, edit (name/pos/desc), and delete update what each command prints on the next invocation without restart.

  **Verify edits without running a turn:**
  - Users should not need `vision`, `prompt`, or an LLM turn to confirm world edits.
  - **`list`** — full world entity overview (agents + objects).
  - **`objects`** / **`agents`** — focused lists when working on one category.
  - **`state`** — active-agent context (memory, turn count, few-shots); complements but does not replace the listing commands.

- [x] Integration with stepper and world

  **Agreed Decision for V0.1:**

  **Command ownership:**
  - All create/edit/delete commands and all listing commands (`objects`, `agents`, `list`) operate at **world level**, regardless of which agent is currently active.

  | `edit-agent` | `do_edit_agent` | |

  - `objects` iterates `world.get_objects()`.

  **Observability updates:**
  - **`objects` (`do_objects`)** — **required.** Lists all objects (id, name, position).
  - **`agents` (`do_agents`)** — **required.** Lists all agents (id, name, position, active marker). Enhance existing V0 handler to include ids.
  - **`list` (`do_list`)** — **required.** Lists everything: agents block then objects block, reusing the same formatting helpers as `agents` and `objects`.
  - `do_state` — enhance object listing to include ids (consistent with `objects` lines); `state` remains active-agent context (memory, turns, few-shots), not a replacement for `list`.
  - `vision` / `prompt` — not required for verifying edits; `list` / `objects` / `agents` cover world inspection.

  **World layer:**
  - Use and extend existing `add_object`, `add_agent`, `get_object_by_id`, etc.
  - Edit/delete logic may call into `world_edit` helpers; invalidation uses `World.invalidate_entity_knowledge()` from Section 1.

### V0.1 perception extension (passive + detailed descriptions)

- [x] Two-layer object descriptions

  **Agreed Decision for V0.1:**

  Extend the `Object` model with a second description field:

  | Field | Model attribute | CLI keyword | Role |
  |-------|-----------------|-------------|------|
  | **Passive** | `passive_description` | `pdesc` | Glance-level text visible without `look` |
  | **Detailed** | `description` | `desc` | Full text revealed by `look`; drives `[?]` when hidden |

  **Passive vision rules (`format_object_vision_desc`):**

  | Passive | Detailed | Never examined | Current (`looked_at`) | Stale (`ever_looked`, invalidated) |
  |---------|----------|----------------|----------------------|-------------------------------------|
  | — | — | *(empty)* | *(empty)* | N/A |
  | — | yes | `[?]` | `desc` | `[?] [changed]` |
  | yes | — | `pdesc` (no `[?]`) | `pdesc` | N/A (no detailed to invalidate) |
  | yes | yes | `[?] {pdesc}` | `desc` | `[?] [changed] {pdesc}` |

  **Key behaviors:**
  - `[?]` only appears when a **detailed** description exists and is hidden or stale.
  - Stale examined knowledge uses **`[?] [changed] {passive}`** (not the old name-based changed sentence).
  - If detailed is empty, `look` returns a no-detail message and does **not** update memory.
  - **`desc` edit** → `invalidate_entity_knowledge()` (objects and agents).
  - **`pdesc` / `name` / `pos` edit** → no invalidation.

  **Initial world (`create_initial_world`):**
  - Remove sign pre-mark (`mark_looked_at` on startup).
  - **Ball:** detailed only → starts as `[?]`.
  - **Sign:** passive `"A simple wooden sign on the wall."` + detailed sign text → starts as `[?] A simple wooden sign on the wall.`

  **Commands:**
  - `create-object` / `edit-object` accept optional `pdesc "..."` alongside `desc "..."`.
  - Example: `create-object name "Sign" pdesc "A sign." desc "It reads: ..." at 2,4`

  **Prompt / few-shots:**
  - Update system rules and examples to reference passive lines, `[?]`, and `[?] [changed]`.
  - Example 4 stale ball line: `Ceramic Ball (obj_ball_01), (2, 2) - [?] [changed]`

  **Success criteria (extension):**

  | # | Test | Expected result |
  |---|------|-----------------|
  | 1 | Fresh world `vision` / `prompt` | Ball `[?]`; sign `[?] {passive}`; agent memory empty |
  | 2 | `look obj_sign_01` | Detailed sign text in vision; memory updated |
  | 3 | Object with `pdesc` only | Passive shown, no `[?]`; `look` does not mark memory |
  | 4 | After `desc` edit on examined object | `[?] [changed] {passive}` (or `[?] [changed]` if no passive) |
  | 5 | `edit-object` with `pdesc` only | No invalidation |
  | 6 | Existing pytest suite | All tests pass with updated expectations |

## Section 3: Multi-agent support

- [x] Confirm Agent and Memory models (as shipped)

  **Agreed Decision for V0.1 (final):**

  Each `Agent` carries its own `Memory`. The shipped `Agent` model adds observable text layers and action visibility:

  **Shipped `Agent` fields:**
  - `id`, `name`, `position`, `memory: Memory`, `last_action`
  - `passive_description` (`pdesc`) — glance text for other agents' passive vision
  - `description` (`desc`) — detailed observable text; hidden behind `[?]` until `look`
  - `personality` — private LLM instructions in **this agent's prompt only**; never in vision or look
  - `passive_result` — third-person summary of this agent's most recent observable action (for other agents' vision)

  **`Memory` (unchanged structure, extended use):**
  - Recent turns (`TurnRecord` list, max 10) + `looked_at` + `ever_looked`
  - `looked_at` / `ever_looked` track **objects and other agents** (by entity id)

  **Isolation rules:**
  - One agent looking at an object or another agent does **not** update another agent's memory.
  - Turn history is private per agent.
  - No shared memory, beliefs, or relationship tracking.

  **Implementation note:** Tests confirm two agents do not leak observations or history (see success criteria).

- [x] Update World model for multiple agents

  **Agreed Decision for V0.1:**

  `World` already has `agents: list[Agent]` and `add_agent()`. Extend with thin lookup helpers; keep backward compatibility for single-agent code and tests.

  **`get_agent()` (unchanged behavior):**
  - Returns `agents[0]` if any exist, else `None`.
  - Update docstring only: "Returns the first agent (backward compatibility for V0 / single-agent tests). For multi-agent, prefer `get_agents()` or lookup helpers."

  **New helpers on `World`:**

  | Method | Behavior |
  |--------|----------|
  | `get_agents() -> list[Agent]` | Return all agents (a copy of the list is acceptable). |
  | `get_agent_by_id(agent_id: str) -> Optional[Agent]` | Linear scan by `agent.id`, same pattern as `get_object_by_id`. |
  | `get_agent_by_name(name: str) -> Optional[Agent]` | Case-insensitive match on `agent.name`. |

  **Unchanged:**
  - `create_initial_world()` — still one "Explorer" agent + ball + sign (passive + detailed; no pre-mark).
  - `remove_agent()` — specified in Section 2 (implementation during Section 2/3 code pass).

  **Out of scope for V0.1:** agent-at-position queries, movement blocking between agents, speak targeting, relationships.

- [x] Enhance ManualStepper for multi-agent

  **Agreed Decision for V0.1:**

  Most multi-agent stepper behavior **already exists** in V0 (`self.agents` dict, `default()` name dispatch, `_run_llm_turn_for_agent`). Section 3 adds `switch`, `run`, dict-sync helpers, reserved-name validation, and small `state` polish — it does **not** redefine Section 2 listing commands (`objects`, `agents`, `list`).

  **Already working (verify during implementation, do not rewrite):**
  - Typing an agent's name (e.g. `Explorer`, `Goblin`) → LLM turn for **that** agent only.
  - `_run_llm_turn_for_agent`: sets `self.agent` at start (so `vision`/`state` match even on LLM failure); `build_prompt` → LLM → `step_turn` → increment `session_turn` → `log_turn`.
  - `vision`, `state`, `prompt`, manual `step` → operate on `self.agent` (active agent).
  - Stepper commands are **case-insensitive** (`Run` and `run` both invoke `do_run`).

  **New command: `run`**
  - Runs an LLM turn for the **currently active** agent (no arguments).
  - Primary workflow after `switch`: e.g. `switch Goblin` then `run`.
  - Same execution path as typing the active agent's name.

  **New command: `switch <name>`**
  - Sets `self.agent` only — **no LLM call, no turn consumed, no turn number increment**.
  - Name match is case-insensitive (same as name dispatch).
  - Example: `switch Goblin` → `Active agent: Goblin (agent_goblin_01) at (0, 3)`
  - Error if not found: clear message pointing user to `agents` or `list`.
  - **Typing an agent's name still always runs an LLM turn** — `switch` is separate; do not merge switch into `default()` dispatch.

  **Reserved agent display names (`src/stepper_commands.py`):**
  - `create-agent` and `edit-agent name` reject names that collide with stepper commands (case-insensitive; hyphen/underscore variants).
  - Reserved names are derived from `ManualStepper`'s `do_*` handlers at runtime (plus `?` for help) so new commands are covered automatically.

  **Dict sync helpers on `ManualStepper` (used by Section 2 agent mutations):**

  | Helper | When |
  |--------|------|
  | `_register_agent(agent)` | After `create-agent` — add to `self.agents[name.lower()]`. |
  | `_unregister_agent(agent)` | After `delete-agent` — remove from dict; fix `self.agent` if needed. |
  | `_rename_agent_in_dict(old_name, agent)` | After `edit-agent name` — remove old key, add new. |
  | `_resolve_agent_by_name(name) -> Optional[Agent]` | Shared by `switch` and `default()` name dispatch; reads `self.agents` dict, rebuilds from `world.agents` on cache miss. |
  | `_rebuild_agents_dict_from_world()` | Called from `__init__` and as safety net in `_resolve_agent_by_name`. |

  **Active agent rules (consistent with Section 2):**
  - `create-agent` does **not** change active agent.
  - LLM turn (typing name or `run`), or `switch` **does** change active agent (`run`/name dispatch set active at start of `_run_llm_turn_for_agent`).
  - `delete-agent` on active agent → fallback to `world.agents[0]` after removal; print `Active agent: …`.

  **`state` polish:**
  - Show active agent **id** as well as name: `Active agent: Explorer (agent_01) at (1, 1)`.
  - Show `session_turn` as console/log label only (not per-agent `TurnRecord.turn_number`).
  - Show `ever_looked` (Section 1) for multi-agent debugging.

  **Documentation:**
  - Stepper `intro` and help must explain: **`switch`** = change active agent without a turn; **`run`** = LLM turn for active agent; **typing agent name** = LLM turn for that agent.

- [x] Agent turns and observability

  **Agreed Decision for V0.1:**

  **Per-agent prompt and actions:**
  - `build_prompt(agent, world, ...)` uses `agent.name`, `agent.personality`, `agent.description` (self-knowledge), `agent.position`, `agent.memory`, and `build_passive_vision`.
  - `step_turn(agent, world, ...)` only mutates the passed-in agent; updates `passive_result` when the action is observably successful.
  - `build_passive_vision(agent, world)` lists **all objects** and **all other agents** (not self). Other-agent lines use the same `[?]` / `[?] [changed]` rules as objects, plus appended `passive_result` when set.

  **Per-agent turn numbering (important):**
  - `TurnRecord.turn_number` must be **per-agent sequential** (1, 2, 3, … within that agent's memory).
  - When recording a turn: `turn_number = agent.memory.turn_count + 1` (or equivalent before append).
  - Do **not** use a single global counter for `TurnRecord.turn_number` — global numbering leaves gaps in per-agent history when agents alternate (e.g. Turn 1, Turn 3 for Explorer), which can confuse the model.
  - `self.session_turn` on `ManualStepper` increments for **console/log labels only** — not stored in `TurnRecord`. Increment **after** successful `step_turn` (both LLM and manual `step` paths).

  **Memory / perception isolation:**
  - Each agent's `looked_at`, `ever_looked`, and turn history are independent.
  - Agents share the same object grid; multiple agents may occupy the same tile (no blocking).
  - Entity **`desc`** invalidation (Section 1) applies per-agent — agents who never looked still see never-examined vision (`[?]` or `[?] {passive}`).

  **Logging:**
  - Turn logs should identify which agent acted (LLM path already prints agent name; manual `step` should log active agent name too).

  **Out of scope (unchanged from V0.1 scope):**
  - No speak **targeting** (speech is room-wide via `passive_result`).
  - No relationships, beliefs, or agent-driven world edits.
  - No automatic turn sequencing — human decides order by typing agent names, `run`, or using `switch` + `step` / `run`.

## Section 4: Observable actions and speak validation (shipped with v0.1.0)

- [x] Third-person action visibility (`passive_result`)

  **Agreed Decision for V0.1 (as shipped):**

  Successful observable actions set the acting agent's `passive_result`, appended on that agent's line in **other** agents' passive vision:

  | Action | `passive_result` example |
  |--------|--------------------------|
  | `move` (success) | `Goblin moves north.` |
  | `speak` | `Goblin says: "Hello."` |
  | `look` | `Explorer examines Goblin.` |

  - Failed moves do **not** update `passive_result`.
  - When the turn includes `confidence` / `emotion`, append `(confidence: …, Emotion: …)` to `passive_result`.
  - Implemented via `ActionOutcome` in `src/action_outcome.py`; `step_turn` applies mood suffixes.

- [x] Speak content limits

  - Ellipsis runs (`...`) treated as pauses, not extra sentence boundaries (`count_speak_sentences`).
  - Pure dialogue enforced by **prompt only** — no runtime rejection of `*`, `_`, or parentheses in speak content.
