# Version 0.3.0 Changelog

**Purpose:**  
Living record of what ships in **V0.3.0** and why ‚Äî **not** a pre-implementation checklist. Each section corresponds to an incremental release slice (e.g. `0.3.0a`, `0.3.0b`) so we can implement and review agilely without a full spec up front.

**Baseline:** Builds on **V0.2.5** (`v0.2.5`) ‚Äî single compound LLM call, pluggable memory modules, Passive Vision‚Äìfirst prompt, compound turn execution (move ‚Üí look ‚Üí turn action).

**Overall V0.3.0 focus:** Turn Realm-Fabric into a **reusable engine** ‚Äî a pip-installable library with a stable session API, serializable world state, and pluggable game profiles (prompt templates, default worlds). The CLI becomes a thin reference client. **No web UI in 0.3.0** ‚Äî that ships in **V0.3.1** as an example project built on this engine.

See also: [ROADMAP.md](ROADMAP.md), [v0.2.5-changelog.md](v0.2.5-changelog.md), [LONG_TERM_GOALS.md](../LONG_TERM_GOALS.md).

## Why V0.3.0 before web

Realm-Fabric should work as an **engine** others can build on (D&D NPC tools, slice-of-life RP, strategy games, etc.). A web GUI is the first *consumer*, not the core.

Shipping the engine first avoids:

- Web-specific APIs baked into `main.py` that never migrate into the library
- Duplicate world state (Python sim vs frontend)
- Prompt logic trapped in one monolithic `build_compound_prompt()` that every game must fork

**V0.3.0** proves the library contract with pytest + CLI. **V0.3.1** proves it with a browser.

## 0.3.0a ‚Äî Session API

**Status:** ‚úÖ **Implemented** in code. CLI refactor to use `Session` ‚Üí **0.3.0d**.

**Goal:** One object owns runtime state and exposes the operations every client needs (CLI today, HTTP handlers in 0.3.1).

### Shipped API (`src/session.py`)

| Type / method | Detail |
|---------------|--------|
| **`Session.from_default()`** | Demo world (Explorer, ball, sign) |
| **`Session(world, active_agent_id=‚Ä¶)`** | Custom world + active agent |
| **`get_active_agent()` / `get_agent(name_or_id)`** | Resolve by id or case-insensitive name |
| **`set_active_agent(name_or_id)`** | ‚Üí `SessionResult` |
| **`run_compound_turn(turn, agent_id=‚Ä¶)`** | ‚Üí `TurnResult` with `TurnRecord`; increments `session_turn` |
| **`build_prompt(name_or_id=‚Ä¶)`** | Compound prompt (delegates to `build_compound_prompt` until **0.3.0c**) |
| **`run_command(line)`** | World edits + listings ‚Üí `CommandResult` |

### Command coverage

`create-object`, `edit-object`, `delete-object`, `create-agent`, `edit-agent`, `delete-agent`, `list`, `objects`, `agents`, `effects`, `memory-modules`.

Not in `run_command` (use dedicated methods): `step-compound`, LLM turns (stay in CLI until **0.3.0d** / later `run_llm_turn`).

### Tests

- `tests/test_session.py` ‚Äî **13 tests** including web-handler-style flow (create ‚Üí move ‚Üí interact)
- Full suite green at implementation time

## 0.3.0b ‚Äî Area snapshot / serialization

**Status:** ‚¨ú **Planned**

**Goal:** Clients (web, save/load later) consume **JSON-friendly state**, not dataclass internals.

### Snapshot shape (draft)

```json
{
  "grid": { "min_x": 0, "max_x": 4, "min_y": 0, "max_y": 4 },
  "area_description": "...",
  "active_agent_id": "agent_explorer_01",
  "agents": [
    {
      "id": "agent_explorer_01",
      "name": "Explorer",
      "position": [1, 1],
      "passive_result": "",
      "memory_module": "recent_turns"
    }
  ],
  "objects": [
    {
      "id": "obj_ball_01",
      "name": "Ceramic Ball",
      "position": [2, 2],
      "actions": ["kick"]
    }
  ]
}
```

### Scope

| In | Out (later) |
|----|-------------|
| `snapshot()` from live `Area` | Full round-trip `load_snapshot()` ‚Üí `Area` (optional stretch) |
| Stable field names for 0.3.1 web | Encrypted saves, multiplayer sync |
| Omit LLM-only fields from snapshot (`personality`) or gate behind `include_private` | Streaming partial updates |

Perception text (`build_passive_vision`) may be a **derived field** on snapshot for active agent (`passive_vision: str`) so web doesn‚Äôt reimplement `[?]` rules.

### Tests

- Snapshot round-trip equality on ids, positions, names
- `personality` not in public snapshot by default
- After turn, snapshot reflects new positions and `passive_result`

## 0.3.0c ‚Äî GameProfile + prompt templates

**Status:** ‚¨ú **Planned**

**Goal:** Projects customize **prompt prose and scenario defaults** without forking core sim code ‚Äî similar in spirit to SillyTavern prompt editing, but structured.

### Concepts

| Type | Role |
|------|------|
| **`PromptContext`** | Core-built facts: passive vision, memory block, look targets, interact list, agent identity, room text |
| **`PromptTemplate`** | String or file with slots: `{{passive_vision}}`, `{{memory}}`, `{{rules}}`, `{{output_format}}`, ‚Ä¶ |
| **`GameProfile`** | Bundles templates, few-shot examples, default world factory, compound-turn rules, schema id (`AgentCompoundTurn` for now) |

### Reference profile

Ship **`profiles/default_compound/`** (or in-code default) that reproduces today‚Äôs 5√ó5 room compound prompt **byte-for-byte in behavior** ‚Äî migration proof.

### Design rules

- Core **always** builds `PromptContext` from sim state (same as today‚Äôs `build_passive_vision`, `render_prompt_block`, etc.).
- Profile **never** computes game rules (range checks, interact validity) ‚Äî only text layout and instructions.
- `build_compound_prompt(agent, world)` becomes `profile.build_prompt(context)` or `session.build_prompt(agent_id)`.

### Explicitly not in 0.3.0c

- GUI prompt editor
- Multiple turn schemas (40k-style phases) ‚Äî `AgentCompoundTurn` remains the only schema; generalize later if needed
- Profile marketplace / hot-reload in production

### Tests

- Default profile prompt contains `Passive Vision:`, `Memory:`, `move_target`
- Custom test profile with minimal template still receives injected context blocks
- Profile default area factory creates equivalent initial layout to `create_initial_area()`

## 0.3.0d ‚Äî CLI refactor

**Status:** ‚¨ú **Planned**

**Goal:** `ManualStepper` is a **reference client** of `Session`, not the owner of simulation state.

### Changes

- `ManualStepper.__init__` creates `Session.from_profile(default_profile())` (or equivalent)
- `do_run`, `default` (agent name), `step-compound`, world-edit commands delegate to `session.*`
- `do_vision` / `do_prompt` read from session + profile
- File logging hooks unchanged in spirit; session may accept optional log callbacks

### Success criteria

- All existing CLI workflows work (manual steps, `run`, `switch`, world edit, `memory-modules`, ‚Ä¶)
- No behavioral regressions in compound turn execution or memory modules
- `main.py` shrinks ‚Äî orchestration moves to `Session`

### Tests

- Existing stepper tests (`test_stepper.py`) green
- New session tests cover paths previously only reachable via CLI
