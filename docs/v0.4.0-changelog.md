## Why V0.4.0

V0.3.x made realm-studio feel like a tabletop session in **one room**. V0.4.0 adds the next layer of “place” and “motion”:

1. **Smarter movement** — LLM can say “go to the ball” (`obj_ball_01`) not only `"2,2"`; optional **move speed** makes travel cost turns on larger grids.
2. **Multiple areas** — one session, many rooms; GM edits and views one area at a time; agents live in exactly one area each.
3. **Connectors** — doors, ladders, and paths as normal object actions with a `**move_area`** effect (no special-case door code).

**Deferred from old ROADMAP “V0.4”:** swappable turn schemas / profile-specific executors → likely **V0.5+** unless pulled forward. **Multiplayer** remains in [LONG_TERM_GOALS.md](../LONG_TERM_GOALS.md).

## Movement — design (0.4.0a–b)

### Today

- `AgentCompoundTurn.move_target` must parse as `**x,y`** (`parse_coordinate_target`).
- `move()` **teleports** the agent to that cell in one nav phase step.
- No pathing, blockers, or per-agent speed.

### Target: two move target kinds


| Form       | Example                   | Resolves to                                         |
| ---------- | ------------------------- | --------------------------------------------------- |
| Coordinate | `"2,3"`, `"(2, 3)"`       | Tile `(2, 3)` in agent’s **current area**           |
| Entity id  | `obj_ball_01`, `agent_02` | That entity’s **current position** in the same area |


Invalid id, entity in another area (post-0.4.0c), or out-of-bounds resolved tile → `ERR:INVALID_TARGET` (fail clearly; do not silently stay put).

Position is resolved **at move time** (start of nav phase).

### `move_speed` (0.4.0b)

New field on `**Agent`**:


| Field        | Type         | Default | Meaning                                              |
| ------------ | ------------ | ------- | ---------------------------------------------------- |
| `move_speed` | `int | None` | `None`  | Max movement steps toward target this turn (5e grid) |



| `move_speed` | Behavior                                                                                                |
| ------------ | ------------------------------------------------------------------------------------------------------- |
| `**None`**   | **Unlimited** — same as today (teleport to target tile). Preserves V0.3 parity and existing tests/demo. |
| `1`, `2`, …  | Walk up to N steps toward target along a deterministic path                                             |


**Pathing (v1) — D&D 5e grid rules:** no tile blockers. Each step costs **1** whether **orthogonal or diagonal** (PHB: diagonals are 5 ft, same as straight — no “every second diagonal costs double” variant).

Minimum steps to reach target tile: **Chebyshev distance** `max(|dx|, |dy|)`.

**Greedy step order (deterministic):** while steps remain and not on target tile:

1. If both `dx` and `dy` are non-zero → **diagonal** step (reduce both)
2. Else if `dx` non-zero → **orthogonal** along x
3. Else if `dy` non-zero → **orthogonal** along y

Stop when `move_speed` steps are exhausted or target tile reached.

Example: from `(0,0)` to `(3,1)` with speed 2 → `(1,1)` then `(2,1)` (“towards”); speed 3 → reaches `(3,1)`.

**Result / passive wording:**


| Outcome                   | Agent `result`                                  | `passive_result`          |
| ------------------------- | ----------------------------------------------- | ------------------------- |
| Target tile reached       | `You moved to (x, y).` or `You reached {name}.` | `{agent} moves to …`      |
| Partial (speed exhausted) | `You moved towards {target}.`                   | `{agent} moves towards …` |
| Already on target tile    | unchanged (`You are already at …`)              | unchanged                 |


Include **final position** in passive text so witnesses and memory stay accurate.

**CLI / snapshot:** `create-agent` / `edit-agent` accept `move-speed N`; empty or omit → `None`. Expose `move_speed` on agent dict in snapshot (null = unlimited).

**Prompt:** Move section lists coordinates **and** in-area entity ids with positions (similar to look/interact lists). Optional later: one-line path hint for LLM when speed is limited.

### Schema change (0.4.0a)

Relax `AgentCompoundTurn.move_target` validator: accept coordinate **or** id matching `agent_`* / `obj_`* (same id rules as look/interact).

## Multi-area session — design (0.4.0c1–c2)

### Today

- `Session.area` — single `Area`.
- All agents and objects live in that area.
- `snapshot()` → one grid + entities.

### Target model (0.4.0c1)

```text
Session
  areas: dict[str, Area]           # e.g. "room", "hall"
  agent_area: dict[str, str]      # agent_id → area_id  (or Agent.area_id)
  active_area_id: str             # GM / CLI / realm-studio edit scope
  active_agent_id: str            # unchanged
```

**Scoping rules (v1):**


| Operation                                     | Scope                                       |
| --------------------------------------------- | ------------------------------------------- |
| Compound turn, passive vision, interact, move | Agent’s **current area** only               |
| `run_command` create/edit/delete              | **Active area** only                        |
| `emit_area_event`                             | **Active area** only (room-local GM events) |
| Cross-area passive vision                     | **Out** — agents only perceive their area   |


**Area transfer (internal API):** used by `move_area` effect and future commands — remove agent from source area, validate destination bounds, set position, register in destination area, update `agent_area`.

### Snapshot shape (v1 — full session)

One response includes **all areas** (fine for 2–5 demo rooms; extensible later):

```json
{
  "session_turn": 3,
  "active_agent_id": "agent_01",
  "active_area_id": "room",
  "areas": {
    "room": {
      "grid": { "min_x": 0, "max_x": 4, "min_y": 0, "max_y": 4 },
      "area_description": "...",
      "objects": [],
      "recent_events": []
    },
    "hall": { "...": "..." }
  },
  "agents": [
    {
      "id": "agent_01",
      "name": "Explorer",
      "area_id": "room",
      "position": [1, 1],
      "move_speed": null,
      "appearance": "tokens/explorer.svg"
    }
  ]
}
```

Per-area agent/object lists may be **denormalized** into each area block for grid rendering, or derived client-side from global agent list — pick one in implementation; document in 0.4.0c1.

**Future (out of 0.4.0):** `GET /api/state?area_id=hall` for lazy load when worlds grow large.

### realm-studio (0.4.0c2)


| Piece                | Detail                                                                               |
| -------------------- | ------------------------------------------------------------------------------------ |
| **Area dropdown**    | Toolbar — selects `active_area_id`; grid + edits + emit event scoped to that area    |
| **Agents elsewhere** | Not drawn on current grid; optional sidebar note (“Goblin — hall”)                   |
| **API**              | `POST /api/active-area` `{ "area_id": "hall" }` or include in existing state refresh |
| **Backward compat**  | Single-area profiles still work — one entry in `areas`                               |


### Demo profile

Extend default (or add `two_room` profile): **room** + **hall**, door object in room with `move_area` (wired in **0.4.0d**).

## Object effects — design (0.4.0d)

### Today

```python
ObjectAction.effects: list[str]   # e.g. ["random_move_self"]
apply_effects(area, agent, obj, effect_names)
```

Effect handlers receive `(area, agent, obj)` only. **No parameters.** Multi-area transfer impossible.

### Target: parameterized effects

Extend `**ObjectAction`** so an action can carry effect **config** (not only names):

```python
# Conceptual — exact dataclass TBD in implementation
effects: list[EffectSpec]

@dataclass
class EffectSpec:
    name: str
    params: dict[str, str] = field(default_factory=dict)
```

**CLI (create-object / add-action):**

```text
action walk_through range 0 effect move_area dest-area hall dest-at 1,1
result "You walk through the door." passive "{actor} walks through the door."
```

```text
action climb range 0 effect move_area dest-area roof dest-at 2,2
result "You climb up." passive "{actor} climbs the ladder."
```

Same object can expose **different** actions pointing to **different** destinations (door vs ladder on one tile stack).

### `move_area` registry handler

```python
def _move_area(ctx: EffectContext, agent: Agent, obj: Object) -> None:
    # params: dest_area, dest_at (from EffectSpec)
    # ctx.session.transfer_agent(agent.id, dest_area, x, y)
```

`**EffectContext**` (or pass `Session` directly): gives handlers access to **all areas** and transfer API introduced in 0.4.0c1.

**v1 rules:**

- Validate destination area exists and `(x, y)` in bounds
- Agent must be in source area (the area where interact ran)
- Witnesses in **source** area get normal interact passive result; destination area agents do not auto-see arrival (v1)
- Object is **not** consumed unless paired with `delete_self`

**Backward compat:** existing actions with `effect random_move_self` (name only, empty params) keep working.

**Notes**

- Prefer engine primitives in `src/` with tests; realm-studio stays a thin HTTP/UI consumer.
- Keep `**move_speed=None`** default so existing scenarios and tests opt into pathing explicitly.
- Snapshot returns **all areas** in v1; shape should allow per-area API later without breaking clients.
- Effect registry should stay **data-driven** (`EffectSpec` params) so the next effect (e.g. `teleport_self`, `spawn`) does not require another one-off schema hack.
